# 特征字典

**项目名称：** 百度广告点击率预测与投放优化分析  
**文档阶段：** 第二阶段 — 特征工程（第一版）  
**数据基础：** 清洗后 Parquet / SQLite 表 `train_events`、`test_events`

> 说明：本文档为特征字典第一版，用于梳理原始字段含义及后续特征工程设计方向。  
> 部分衍生特征尚未在代码中全部实现，实际特征工程完成后将继续更新本文档。  
> 文中不涉及具体 CTR 数值或字段基数统计，相关结果以 EDA 与 SQL 分析输出为准。

---

## 1. 目标变量

| 字段名 | 数据类型 | 含义 | 用途 | 是否作为模型输入 |
|--------|----------|------|------|------------------|
| `click` | 整型（0/1） | 用户是否点击广告。0 表示未点击，1 表示点击 | 训练集标签，用于监督学习；CTR 分析的核心因变量 | **否**（仅作标签，不能泄漏到特征中） |

**补充说明：**

- `click` 仅存在于训练集（`train.csv` / `train_events`）。
- 测试集（`test.csv` / `test_events`）**不包含** `click` 字段。
- 清洗后 `click` 转为可空 Int8，全量验收结果显示取值仅为 0 或 1（详见 `outputs/15_processed_validation_report.txt`）。

## 3. 时间衍生特征

以下特征为**计划构建**的衍生字段，主要从 `hour` / `hour_dt` 提取，用于描述曝光发生的时间规律。

| 特征名 | 生成来源 | 含义 | 数据类型 | 状态 |
|--------|----------|------|----------|------|
| `hour_of_day` | `hour` 或 `hour_dt` | 一天中的第几小时，取值 0—23 | 整型 | 计划中（EDA 脚本已从 `hour_dt` 提取并验证） |
| `day` | `hour_dt` | 日期，如 2014-10-21 | 日期 / 字符串 | 计划中 |
| `day_of_week` | `hour_dt` | 星期几，取值 0—6 或 1—7 | 整型 | 计划中 |
| `is_weekend` | `day_of_week` | 是否为周末，0/1 | 整型 | 计划中 |

**设计目的：** 捕捉不同时间段、不同日期的点击行为差异，支撑 CTR 时间维度分析与建模。

## 4. 类别频次特征

以下特征为**计划构建**的频次类特征，用于描述某类别值在训练数据中的出现次数或出现频率，反映「热度」或「常见程度」。

| 特征名 | 生成来源 | 含义 | 数据类型 | 状态 |
|--------|----------|------|----------|------|
| `site_id_freq` | `site_id` | 该 `site_id` 在训练集中的出现次数或频率 | 数值型 | 计划中 |
| `site_category_freq` | `site_category` | 该网站类别在训练集中的出现次数或频率 | 数值型 | 计划中 |
| `app_id_freq` | `app_id` | 该 `app_id` 在训练集中的出现次数或频率 | 数值型 | 计划中 |
| `app_category_freq` | `app_category` | 该应用类别在训练集中的出现次数或频率 | 数值型 | 计划中 |
| `device_model_freq` | `device_model` | 该设备型号在训练集中的出现次数或频率 | 数值型 | 计划中 |

**设计目的：** 对高基数类别字段做压缩表达，避免直接 one-hot 导致维度过高，同时保留类别流行度信息。

**注意：** 频次统计应**仅基于训练集**计算，再映射到 train/test，避免使用 test 标签或全量含 test 的统计造成泄漏。

## 5. 交叉特征

以下特征为**计划构建**的组合特征，用于刻画两个维度之间的联合效应。命名与项目 EDA / SQL 交叉分析保持一致。

| 特征名 | 组合来源 | 设计目的 | 状态 |
|--------|----------|----------|------|
| `banner_device_cross` | `banner_pos` × `device_type` | 分析不同广告位在各设备类型上的点击差异，对应 EDA `11_eda_banner_device.py` | 计划中 |
| `hour_banner_cross` | `hour_of_day` × `banner_pos` | 分析不同时间段各广告位的点击表现，对应 EDA `12_eda_hour_banner.py` | 计划中 |
| `site_device_cross` | `site_category` × `device_type` | 分析不同网站类别中设备类型的点击差异，对应 EDA `13_eda_site_device.py` | 计划中 |

**实现方式（待定）：** 可将两个字段拼接为字符串交叉键（如 `"0_1"`），或分别保留两列供树模型自动组合；具体以实现脚本为准。

## 6. 特征使用注意事项

1. **高基数字段不宜直接 one-hot**  
   `site_id`、`app_id`、`device_id`、`device_ip`、`device_model` 等字段唯一值数量大，直接独热编码会导致特征维度过高、训练成本上升，且容易过拟合。

2. **匿名编码字段不做业务含义解释**  
   `site_category`、`app_category`、`banner_pos`、`device_type` 及 `C1`、`C14`–`C21` 等均为数据中的匿名编码，文档与建模过程中不应随意解释为具体行业、页面位置或设备品牌。

3. **`click` 不能作为输入特征**  
   `click` 是预测目标，仅用于训练标签与 CTR 评估。任何基于 `click` 的统计（如目标编码）必须在交叉验证框架内完成，且不能用到 test 集标签。

4. **test 集没有 `click`**  
   特征工程与编码映射（频次表、目标编码参数等）应仅用训练集拟合，再一致地应用到 test 集。

5. **频次编码与目标编码需防止数据泄漏**  
   - 频次特征：计数应来自训练集，不能包含 test 样本信息。  
   - 目标编码：需在训练折内计算，避免用验证集或测试集标签参与编码。

6. **质量检查字段不得入模**  
   `is_invalid_click`、`is_dup_id_within_chunk` 仅用于数据质量分析，不应作为模型训练特征。

7. **样本标识字段 `id` 的特殊用途**  
   `id` 用于样本追踪与提交文件对齐，一般不参与预测；验收脚本已确认 test 的 `id` 顺序与 `sampleSubmission.csv` 一致。

8. **文档会持续更新**  
   本字典为第一版。待 `data/features/` 下实际特征文件生成后，将补充最终特征名、生成脚本路径及是否入模的确定结论。
